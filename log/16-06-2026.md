Log: Today I wanted to take a step back and read and comprenhed good `nftables`, so here we go:

nftables clasifies, filter, nat, route, transform packets and it's the succesor of iptables. this framework uses tables to contain chains, where in each one contains differents rulesets.

This is the default configuration file that comes when you install nftables:

```
#!/usr/sbin/nft -f

flush ruleset

table inet filter {
    chain input {
        type filter hook input priority filter;
    }

    chain forward {
        type filter hook input priority filter;
    }

    chain output {
        type filter hook input priority filter;
    }
  }
```

I'm going to start from the outside, and that means `table`, defined by: `table <table_address_family> <table_name> { }`

These definitions are from [Red Hat documentation](https://docs.redhat.com/en/documentation/red_hat_enterprise_linux/8/html/securing_networks/getting-started-with-nftables_securing-networks):

```
ip: Matches only IPv4 packets. This is the default if you do not specify an address family.
ip6: Matches only IPv6 packets.
inet: Matches both IPv4 and IPv6 packets.
arp: Matches IPv4 address resolution protocol (ARP) packets.
bridge: Matches packets that pass through a bridge device.
netdev: Matches packets from ingress. 
```

At this moment then, we have a `table` named **filter** that is going to analize packets matching IPv4 and IPv6. Moving forward, tables contain `chains`. There are two types of chains, `basic chain` is just an entry point for the packet or `regular chain` to use it as a **jump** target. As far as I saw, you can use regular chains to jump packets to another chains create to do a specified job, for example SSH or HTTP. Adding up, chains contains **hooks** such as:

- input: packets whose final destination is the local system
- output: packets generated by the local system  
- forward: packets passing through the system ( like a router)
- prerouting: packets before the routing decision is made
- postrouting: packets after the routing decision, right before leaving the system

Aside from rules, we can set `policies` that after all rules are analized, if no one matches it, then the policy comes in action, is like a default behavior if nothing matches:

type <type> hook <hook> priority <priority> ; policy <policy> ;



Now it's time to write it down some tables. Careful with the next table, if you execute you will lock out from everything but I wanted to do it step by step and working towards a system based in a principle of least privilege:

table inet basic_firewall {
    chain input {
        type filter hook input priority 0; policy drop;
    }
    chain output {
        type filter hook output priority 0; policy drop;
    }
    chain forward {
        type filter hook forward priority 0; policy drop;
    }
}
